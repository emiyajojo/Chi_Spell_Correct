# 金融领域文本纠错系统 — 技术说明文档

本文档从算法设计、数据构建、架构选型与工程约束四个维度，说明本项目的技术方案与实现思路。项目面向金融场景的文本纠错需求，采用 **BERT 序列标注 + SimCSE 语义检索** 的混合架构，在“宁可不改、不能改错”的前提下，兼顾专有实体（如股票名称）与通用文本（错别字、语法）的纠错能力。

---

## 1. 深度数据构建 (Advanced Data Synthesis)

金融领域高质量纠错语料极度稀缺，若仅依赖少量人工标注，模型易过拟合且难以泛化到真实录入错误分布。针对这一问题，本项目在数据层面做了系统化设计。

### 1.1 金融场景噪声注入方案

在对比了多种数据增强方式后，我采用了**面向金融场景的噪声注入**，而不仅是通用的 EDA（同义词替换、随机插入/删除等）。核心思路是：**模拟真实业务中出现的错误类型**，使训练分布更贴近录入、语音转写、OCR 等渠道产生的噪声。

- **同音异字**：结合金融词典与拼音库，对关键词（如“券”“涨”“持仓”）生成同音错写（如“卷”“张”“持仓”），用于拟合输入法、语音识别导致的错误。
- **形近字**：基于字形相似度（偏旁、笔画结构），构造易混淆字对（如“未/末”“已/己”），模拟视觉误判与手写识别错误。
- **拼音首字母缩写**：对部分金融术语或股票简称做“首字母/拼音缩写”形式的错误变体，用于覆盖口语化、简写输入带来的长尾错误。

上述策略均在**金融词典约束**下进行，避免在无关位置注入噪声，保证错误-正确对在语义上仍可对应，便于模型学习“纠错”而非“乱改”。

### 1.2 控制过度纠错的干净数据设计

通过实验发现，若训练集中全部为“错误→正确”样本，模型容易学到“见错就改”的偏置，对本身正确的句子也进行不必要的修改（过度纠错）。为解决这一问题，我**显式构造了约 20% 的“干净数据”作为控制组**：即输入与标签完全一致、无需任何修改的样本。这样做的目的是**强迫模型学习恒等映射**，在“有错则改”与“无错则保持”之间取得平衡，从数据层面缓解过度纠错，与后文提到的 PPL 校验、白名单等工程手段形成互补。

## 2. 检索与生成的双路架构 (Hybrid Architecture)

### 2.1 为何在 BERT 之外引入 SimCSE + Faiss

本系统将纠错拆成两条能力线：

- **专有实体纠错（股票名称等）**：由 **Span-based NER（BERT） + SimCSE 向量检索 + Faiss** 完成。  
  BERT 序列标注擅长**局部拼写错误**和**边界敏感**的实体识别；但对金融场景中大量**长尾、固定短语**（如股票名称、基金产品名），仅靠生成式/序列标注容易受训练集覆盖度限制。因此引入 **SimCSE 无监督/有监督句向量 + Faiss 近似最近邻**：先通过 NER 抽取出“疑似实体片段”，再在标准化实体库中做语义检索，用**检索到的标准表达**替代错误片段。这样，**局部错误由 BERT 管，长尾固定短语由检索库管**，分工明确、可解释性强。

- **通用文本纠错（错别字、语法等）**：由 **MacBERT4CSC** 等 BERT-based 纠错模型完成，在专有实体纠错之后对整句再做一次修正，形成“先实体、后全文”的两阶段流程。

### 2.2 置信度分发与检索优先策略

为避免检索结果不可靠时强行替换、引入新错误，我实现了一个**简单的置信度分发逻辑**：

- 对 NER 抽出的待纠错片段，用 SimCSE 在 Faiss 索引中检索 Top-K 候选，并得到相似度（或距离）分数。
- **当检索到的最高相似度高于设定阈值（如 0.95）时，优先采用检索库中的标准化表达**，认为该片段属于“已知实体写错”的情况。
- 若最高相似度低于阈值，则弱化检索结果的影响（例如仅作候选参考，或回退到编辑距离等规则策略），避免低置信度检索带来的误改。

这一设计在代码中对应为：先判断实体是否已在标准库（如 `stock_dic`）中，若不在则通过 `faiss_search` 获取候选，再结合编辑距离等做排序与选择；阈值与排序策略可根据业务指标（如 F0.5）做调优。

## 3. 过度纠错的工程约束 (Engineering Constraints)

金融场景对纠错系统有明确要求：**宁可漏改，不可错改**。因此除了在数据与模型层面控制过度纠错，还需在推理链路上做防御性编程。

### 3.1 PPL（Perplexity）校验与回滚

在得到模型的修正结果后，增加**语言模型得分校验**：

- 使用统一的语言模型（如与 MacBERT 同源的 BERT 或外部 n-gram/LM）分别计算**修正前**与**修正后**句子的 PPL。
- **若修正后的 PPL 反而高于修正前**，则认为本次修改可能降低了句子流畅度/合理性，执行**回滚**：保留原句，不采纳该次纠错。  
这样可以从统计角度过滤掉一部分“改坏了”的 case，尤其对边界样本有效。

### 3.2 白名单与敏感词保护

针对金融实体（如特定基金产品名、合约代码、专有名词），维护**白名单/敏感词库**：

- 若某片段被识别为待纠错实体，但在白名单中，则**不做替换**，仅做记录或透传。
- 避免将正确但少见的实体名误判为错误并替换成检索库中的其他实体，从而满足“不能改错”的硬约束。

当前实现中，对已在标准股票名称集合（如 `stock_dic`）中的实体直接保留不替换，可视为白名单机制的一种实现；可进一步将白名单扩展到基金、合约等更多类型。

## 4. 性能与评估 (Evaluation & Performance)

### 4.1 评估指标：强调 F0.5

在纠错任务中，**精确率（Precision）比召回率（Recall）更关键**：误改会直接导致业务风险，而漏改往往还可通过人工复核补救。因此评估时我**优先采用 F0.5**（即 $\beta=0.5$ 的 F-score），对 Precision 赋予更高权重：

$$
F_{0.5} = \frac{(1 + 0.5^2) \cdot P \cdot R}{0.5^2 \cdot P + R}
$$

在汇报与调参时，会同时关注检错 F1、纠错 F1 以及 F0.5，以便在“少误改”与“多纠错”之间做业务侧权衡。当前 MacBERT 评估管线中采用 0.8×检错 F1 + 0.2×纠错 F1 的综合分，可与 F0.5 一起作为主指标与辅助指标使用。

### 4.2 推理性能与部署

- **服务形态**：基于 **FastAPI** 封装异步推理接口，支持单句/批量的纠错请求，返回原始文本、纠错结果及耗时等信息。
- **延迟目标**：在 GPU 推理条件下，**单句端到端延迟控制在 50ms 以内**（包含 NER、SimCSE 检索、MacBERT 纠错及后处理）。通过 ONNX 导出与 CUDA Provider 进一步压缩推理时间，并在无 GPU 时回退到 CPU，保证可用性。
- **模型加载**：服务启动时一次性加载 NER、SimCSE、MacBERT 及 Faiss 索引，请求阶段仅做前向与检索，避免重复加载带来的延迟抖动。

## 5. 小结

本项目从**数据构建（噪声注入 + 干净数据控制组）**、**双路架构（BERT 序列标注 + SimCSE/Faiss 检索）**、**工程约束（PPL 校验、白名单）** 以及 **评估与性能（F0.5、FastAPI、低延迟）** 四个方向，系统化地针对金融文本纠错场景做了设计与实现。在“宁可不改、不能改错”的前提下，通过数据与推理链路的双重约束，在保证准确率的同时兼顾召回与线上性能，便于后续按业务需求做阈值与策略的微调。
